# 工程架构原则

本 skill 定义语言无关的架构和工程原则。技术专家 Agent 加载本 skill 后，须将通用原则适配到具体语言的表达方式。

REQUIRED: 加载本 skill 后，在所有设计和实现中遵循下述原则
REQUIRED: 将通用原则适配到具体语言的表达方式（如 Rust 用 trait、TS 用 interface、Python 用 protocol）

---

## 设计原则速查

| 原则 | 核心表达 | 判定要点 | 违规信号 |
|------|---------|---------|---------|
| **KISS** | 按需设计，简洁优先 | 两方案持平选更简单的；三个相似调用不是抽象的理由 | 冗余抽象、过度配置化 |
| **DRY** | 复用类型/逻辑 | 语义不同的巧合相似不必强求统一；内聚 > 提取公共代码 | 重复定义、复制粘贴 |
| **YAGNI** | 仅实现当前需求 | 迭代开发中为已知下一步准备的扩展点属合理预留 | 无用例的扩展点 |
| **SRP** | 模块单一职责 | 无法用 ≤3 条描述概括职责 → 需拆分；变更频率不同 → 不应同模块 | 单文件混合多领域 |
| **OCP** | 扩展开放，修改封闭 | 基础设施层适配器模式属合理预留 | 新功能需改核心模块 |
| **ISP** | 接口精简 | 一个接口一个能力维度，避免 God Interface | 强迫依赖不需要的方法 |
| **DIP** | 依赖抽象 | 跨层交互通过接口而非具体类型；依赖方向单向 | 直接依赖具体实现 |
| **认知对齐** | 目录 = 架构概念 | 3 秒内直觉可达 | 目录与文档术语不一致 |
| **内聚优先** | 相关代码相邻 | 改一功能需改 3+ 目录 → 内聚不足 | 改一功能跨 3+ 目录 |

## 模块通信边界

### 三层模型

| 层级 | 可见性 | 通信方式 |
|------|--------|---------|
| **Internal** | 私有，不导出 | 内部自由使用 |
| **Domain** | 公开接口契约 | 方法调用 + 只读状态查询 |
| **Global** | 跨模块共享 | 只读状态 + 强类型事件 |

### 契约

- **外→内**：只能通过公开接口，禁止直接操作内部状态
- **内→外**：暴露只读状态视图，外部通过订阅获取变化
- **跨模块**：仅系统级通知用事件机制，必须类型安全

禁止跨模块深度引用非入口导出。紧耦合模块组可共享内部类型，但须标记为内部接口。

## 层归属校验

> 在分层架构中，代码放错层是最常见也最隐蔽的架构腐化。本节提供实现前的自检机制。

### 三步校验（实现任何非平凡逻辑前必须执行）

1. **定位目标层** — 确认要修改的文件属于哪个架构层（如：网关层、业务层、基础设施层）
2. **读取层职责定义** — 找到该层的职责文档或代码入口注释，确认"该层负责什么"和"不属于该层的有什么"
3. **逐条比对** — 要实现的逻辑，逐条对照层职责定义。任何一条命中"不属于该层"列表 → 停止，重新确定归属

### 依赖方向审计（添加 import/依赖时必须执行）

- 画出或回忆项目的层依赖方向（通常是单向的：上层 → 下层）
- 每添加一个新依赖，确认方向符合规定
- **向上依赖**（下层依赖上层）→ 必须通过接口反转（依赖倒置）
- **跳层依赖**（如网关层直接依赖基础设施层，跳过业务层）→ 需要明确理由，否则视为违规

### 体量信号触发器

当目标文件/模块出现以下信号时，必须质疑层归属是否正确：

| 信号 | 判定标准 | 行动 |
|------|---------|------|
| 薄层文件过长 | 网关/适配器层的文件超过 ~300 行业务逻辑 | 检查是否有业务逻辑泄漏 |
| 同文件多职责 | 文件承担 >3 个不相关职责 | 按 SRP 拆分，检查拆分后的归属 |
| 大量底层依赖 | 一个文件 import 了 >3 个不同基础设施模块 | 检查是否在做"不该在这一层做的事" |
| 算法/业务逻辑出现在传输层 | 网关/API 层中出现数据处理、状态管理、流程编排 | 迁移到对应的业务层 |

## 文件组织

### 核心原则：目录边界 = 认知边界

开发者读了架构文档，就应该能在目录中找到对应位置。判断标准：3 秒内直觉可达。

### 三原则

1. **目录与架构概念一一对应** — 文档中的层/模块，目录中必须有映射
2. **内聚 > 分类** — 相关代码物理相邻。修改一个功能需改 3+ 目录 → 内聚不足
3. **文件大小是设计信号** — ~700 LOC 以内。超标时看变化方向：频率不同→拆分，同步变化→保持

### 按项目类型确定划分维度

| 项目类型 | 划分维度 | 典型结构 |
|---------|---------|---------|
| 工具类库 | 功能域 | `src/string/` `src/array/` |
| UI 组件库 | 组件域 | `src/button/` `src/modal/` |
| 引擎/后端 | 架构层 | `src/engine/` `src/runtime/` `src/store/` |

**禁止**：版本复制文件、桶命名（utils/helpers/misc）、同语义目录并存

## 错误处理

### 分层策略

- **库层**：定义具体错误类型，错误变体反映可恢复的失败模式
- **应用层**：聚合底层错误，保留完整错误链（附加上下文信息）
- **模块边界**：每个模块定义自己的错误类型，错误在边界转换，禁止跨层透传

### 基本规则

- "值可能不存在" 用 Optional 类型表达
- "操作可能失败" 用 Result 类型表达
- 二者不混用

**禁止**：业务逻辑路径中用异常/panic 控制流

**特例**：
- 断言式操作仅用于"如果失败说明程序本身有 bug"的场景
- 测试代码中的直接断言是可接受的
- 原型验证阶段可宽松，但须在技术债务中记录

## 并发模型

### 优先级

1. **Message passing（channel）** > shared state（锁）
2. 必须用锁时：优先读写锁（读多写少）或互斥锁（写多）
3. 并发容器的 check-then-update 必须用原子 API 完成，禁止 TOCTOU

### 约束

- 无限期阻塞操作必须配合 shutdown 机制，确保进程可退出
- 先检查是否有工作，再获取并发许可——禁止"先 acquire → 再检查"的顺序
- 并发容器选型应统一（同一模块内不混用多种并发原语而无理由）

**特例**：
- 高频热路径（>10K ops/s）可用无锁并发结构，但必须限定在单层内部
- 单线程上下文（如测试辅助）中线程不安全的容器是合理的，但必须标注使用边界

## 测试原则

- 每个模块必须可单元测试；测试验证行为，而非实现细节
- Mock 通过抽象接口实现提供，禁止修改生产代码以适应测试
- 核心功能路径必须经过真实测试验证
- 异步测试中的子任务必须在测试结束前完成
- 阻塞式等待必须包裹超时，防止测试挂起

**特例**：当被测逻辑强依赖 IO（文件、网络）且 mock 成本过高时，允许集成测试使用真实资源，但必须可隔离（临时目录、随机端口等）

## 数据完整性

> 展示数据必须来自权威数据源，禁止兜底/猜测值。

**展示即真相** — 所有展示的数值、状态、计数必须来自权威数据源的实际数据，不得用硬编码默认值或猜测值填充。

**不可用时诚实表达** — 数据加载失败或不可用时，显示"暂无数据"/loading 状态，而非编造值。

**计数准确性** — 列表计数必须与实际加载的数据一致。

**禁止**：用兜底值为展示提供"看起来合理"的值；为缺失数据编造显示文本；使用过时的预加载数据

## 工程决策框架

当面临多种实现方案时，按以下顺序决策：

1. **是否满足需求** — 不满足的方案不进入比较
2. **数据流是否清晰** — 数据流向和生命周期是否直观
3. **静态分析能消灭多少错误** — 类型设计是否排除了无效状态
4. **变更影响范围** — 修改一个需求需要改几个地方
5. **可测试性** — 能否不修改生产代码就完成测试

如果两个方案在前 4 项持平，选更简单的那个（KISS）。

## 实施纪律

**必须**：
- 实现前确认需求的交付定义，不凭印象实现
- 对非显而易见的逻辑添加简要注释——解释 "为什么" 而非 "做什么"
- 结构化日志：键值对形式，而非格式化字符串
- 公共 API 有文档注释，内部实现不需要

**禁止**：
- 绕过设计问题（而非出于明确的性能/语义理由）
- 硬编码本应可配置的值（超时、重试次数、资源限制）
- 为当前不需要的功能预留扩展点（YAGNI）
- 不经讨论简化需求的实现——如果认为可以简化，需提出方案选项并说明 trade-off
